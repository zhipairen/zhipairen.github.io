TCP

TCP传输控制协议：会将数据分段打包传输，并对每个数据包编号控制顺序，传输中出现丢包时，会有重发等机制处理保证数据完整性

三次握手：

第一次握手主机A通过一个标识为SYN标识位的数据段发送给主机B请求连接，通过该数据段告诉主机B希望建立连接，需要B应答，并告诉主机B传输的起始序列号
第二次握手是主机B用一个确认应答ACK和同步序列号SYNC标志位的数据段来响应主机A，一是发送ACK告诉主机A收到了数据段，二是通知主机A从哪个序列号做标记。
第三次握手是主机A确认收到了主机B的数据段并可以开始传输实际数据。

第一次握手主要是确定服务端确认客户端能够发送信号；第二次握手主要是客户端确认服务端能够接收和发送信号；第三次握手主要是服务端确认客户端能够接收信号
四次挥手：

主机A发送FIN控制位发出断开连接的请求
主机B进行响应，确认收到断开连接请求
主机B提出反方向的关闭要求
主机A确认收到的主机B的关闭连接请求

第一次挥手是服务端确认客户端需要断开连接；第二次挥手是客户端确认服务器接收断开请；第三次挥手是客户端确认服务器数据发完，断开连接；第四次挥手是服务端确认客户端断开连接，断开连接。所以如果服务端的数据全部发送完，是没有第三次挥手，直接进入第四次挥手。

Streams

Everything you need to know about streams

概念

Streams(流)是数据的集合-就像数组或字符一样；
不同的处在于流可能不会一次全部处理,而且它们不适合内存；在处理大数据或者外部源的数据chunk(块)的时候，流可以发挥巨大能量。
流不仅能处理大数据，还可以在代码提供可组织性的能力，像管道一样进行组合。
注意：node本身缓存有限制

分类

在node里四种基本的流： Readable(可读)、Writable(可写)、Duplex(双工)、Transform(变换) streams

  • Readable stream是消费数据的源的抽象，eg: fs.createReadStream()
  • Writable stream是可以写入数据的目标的抽象，eg: fs.createWriteStream()
  • Duplex streams是可读和可写的，eg: TCP 套接字(socket)
  • Transform stream基本上是双工流，可用于在写入和读取数据时修改或变换数据。eg:使用gzip压缩数据的zlib.createGzip流。 您可以将变换流视为一个函数，其中输入是可写流部分，输出是可读流部分。 您可能还会听到称为“直通流”的变换流。

    所有流都是EventEmitter的实例。 它们发出可用于读取和写入数据的事件。 但是，我们可以使用管道方法以更简单的方式使用流数据。

pipe方法

以可读的数据源作为输出，以可写的目标作为输入
stream events

流可以直接被事件消费。注： event和pipe两者都能消费流，但不能同时存在
常用的重要流事件：

  • 可读流：data事件(消费数据中)、end事件(没有更多数据供流消费)
  • 可写流： drain事件(可写流能继续接收更多数据的标识)、finish事件

可读流的暂停和流动模式
可读流有两种主要模式：paused(暂停) mode 、 flowing(流动) mode;有时候被简称为 pull 和 push modes.

默认情况下，可读流以暂停模式启动，但是可以很容易变换为流动模式，有时，模式切换可自动发生。

当可读流在paused模式下，可以用 read() 方法按需读取流；但是，在flowing模式下，数据不断流动，必须通过监听事件的方式去消费它。
在flowing模式下，如果没有消费处理数据，实际上数据可能会丢失，也就是我们需要data事件处理。实际上，当增加了data事件处理就可以把暂停模式切换为流动模式，并且移除data事件处理程序的话会将流切换回暂停模式。
如果要在这两种流模式之间手动切换，可以使用resume（）和pause（）方法。

当用pipe处理可读流时，不用担心模式问题，管道pipe会自动管理好它们。

Child Processes

Node.js中的单线程，非阻塞性能非常适合单个进程。 但最终，一个CPU中的一个进程不足以处理应用程序不断增加的工作负载。使用多个进程是扩展Node应用程序的最佳方法。
child_process模块使我们能够通过在子进程中运行任何系统命令来访问操作系统功能。

在Node中创建子进程有四种方法： spawn(), fork(), exec() 和 execFile().注：该模块是EventEmitter的实例
spawn()

每个子进程也获得三个标准stdio流，我们可以使用child.stdin，child.stdout和child.stderr访问它们。
当这些流关闭时，正在使用它们的子进程将发出close事件。 此关闭事件与退出事件不同，因为多个子进程可能共享相同的stdio流，因此一个子进程退出并不意味着流已关闭。
在子进程中，stdout/stderr流是可读流，而stdin流是可写流。(主进程相反)在可读流上，我们可以监听数据事件，该事件将具有命令的输出或执行命令时遇到的任何错误.
子进程以code为 0 退出，表示无错误, code 为 1，表示有错误

shell 语法 和 exec函数
exec函数创建shell, 它缓冲命令生成的输出并将整个输出值传递给回调函数（而不是使用流，这是spawn所做的）.

execFile
如果您需要在不使用shell的情况下执行文件，则execFile函数就是您所需要的。 它的行为与exec函数完全相同，但不使用shell，这使得它更有效。
在Windows上，某些文件无法自行执行，如.bat或.cmd文件。 使用execFile无法执行这些文件，执行它们需要exec或spawn，shell设置为true。

fork()
fork函数是产生节点进程的spawn函数的变体。 spawn和fork之间的最大区别在于，当使用fork时，会为子进程建立通信通道，
因此我们可以使用forked进程上的send函数以及全局进程对象本身来在父进程和fork进程之间交换消息。

对于长计算的函数，可将其fork子进程处理,不阻塞主进程

ChildProcess类

‘close’： 当子进程的 stdio 流被关闭时触发
‘disconnect’: 断开连接
‘error’: 错误
‘exit’: 退出，子进程的 stdio 流可能依然是打开的

subprocess.send(message[, sendHandle[, options]][, callback])

sendHandle 参数可用于将一个 TCP server 对象句柄或 socket 对象传给子进程

当服务器在父进程和子进程之间是共享的，则一些连接可被父进程处理，另一些可被子进程处理。

cluster(集群)

cluster
Node的集群模块不仅提供了利用机器全部CPU能力的开箱即用解决方案，而且还有助于提高节点流程的可用性，并提供以零停机时间重启整个应用程序的选项。
有三种办法来扩展程序：
1、克隆: 通过多次克隆它并让每个克隆实例处理工作负载的一部分（例如，使用负载均衡器），Node内置的cluster模块可以容易实现；
2、分解: 可以通过基于功能和服务对应用程序进行分解来扩展应用程序(eg: 微服务)
3、拆分: 可以将应用程序拆分为多个实例，其中每个实例仅负责应用程序数据的一部分。 此策略通常在数据库中命名为水平分区或分片。 数据分区需要在每个操作之前执行查找步骤，以确定要使用的应用程序的哪个实例。

集群模块可以实现克隆可伸缩性策略，但只能在一台机器上实现；在机器上的多个内核上启用负载均衡；

在Node中，创建一个主进程，主进程fork许多工作进程并管理它们。每个工作进程代表我们要扩展的应用程序的实例。所有传入请求都由主进程处理，主进程决定哪个工作进程应处理传入请求。

主进程的工作很简单，因为它实际上只是使用循环算法(round-robin algorithm)来选择工作进程
Load-Balancing an HTTP Server(负载平衡HTTP服务器)
模拟简单的CPU工作

这些是完全不同的Node.js进程。 这里的每个工作进程都有自己的事件循环和内存空间。

与单个Node进程相比，在机器上创建集群每秒约能够处理181个请求，大概三倍的性能提升。

集群模块只使用了child_process.fork API，这意味着我们在主进程和每个worker之间也有可用的通信通道。
使用cluster.workers访问worker对象列表，读取相关worker信息

假如需要避免多个数据库请求，并将此调用缓存一段时间；

提高服务可用性
当出现实例崩溃时，能重新启动服务器以部署代码(eg: pm2).
使用 process.exit模拟

当工作进程退出时，将使用集群模型对象上的exit事件通知主进程。 我们可以为该事件注册一个处理程序，只需在任何工作进程退出时派生一个新的工作进程。

零停机重启：监听SIGUSR2之类的过程信号，当主进程收到SIGUSR2信号时，这意味着它需要重新启动其工作程序，
但我们希望一次只执行一个这样的工作程序。这只是意味着主进程只应在重新启动当前进程时重新启动它。

共享状态和负载均衡
通过集群设置，我们不能再将内容缓存在内存中，因为每个工作进程都有自己的内存空间。

如果我们需要使用集群设置来缓存内容，我们必须使用单独的实体并从所有工作者读取/写入该实体的API。
此实体可以是数据库服务器，或者如果要使用内存缓存，则可以使用Redis之类的服务器，或者创建一个带有读/写API的专用Node进程，供所有其他工作人员进行通信。

工作进程状态通信

主进程进行身份验证,再发送给工作进程；
通过将这些会话的信息存储在共享数据库或Redis节点中，我们可以简单地在我们拥有的许多工作者之间共享状态。

Event loop
每个 chunk 通常是以 function 为单位，一个 chunk 执行完成后，才会执行下一个 chunk。下一个 chunk 是什么呢？取决于当前 Event Loop Queue 中的队首。
Event Loop Queue 中存放的都是消息，每个消息关联着一个函数，JavaScript Engine 就按照队列中的消息顺序执行它们，也就是执行 chunk。

Job Queue
Job 是 ES6 中新增的概念，它与 Promise 的执行有关，可以理解为等待执行的任务；Job Queue 就是这种类型的任务的队列。JavaScript Runtime 对于 Job Queue 与 Event Loop Queue 的处理有所不同。

相同点：都用作先进先出队列
相异点：
每个 JavaScript Runtime 可以有多个 Job Queue，但只有一个 Event Loop Queue
当 JavaScript Engine 处理完当前 chunk 后，优先执行所有的 Job Queue，然后再处理 Event Loop Queue
ES6 中，一个 Promise 就是一个 PromiseJob，一种 Job。每一个promise都是独立的promiseJob(JOb Queue),其then之后的回调会存储在该job中
